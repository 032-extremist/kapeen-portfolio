Building RedCart: what a full-stack store actually takes

When I started RedCart, I thought the hard part would be the storefront — product grids, a cart, checkout. It wasn't. The hard part was everything underneath it that the user never sees: the data model, the boundary between backend and frontend, and the fact that a store has to work the same way every single time, not just on a good day.

Starting with the data model

I used Prisma as the ORM layer on top of a TypeScript backend, and the first real decision wasn't a UI decision at all — it was deciding what a "product," an "order," and a "user" actually are, and how they relate. Get that wrong early and every feature after it gets harder to build. Prisma's schema file forced me to write that down explicitly before I could write a single API route, which turned out to be a good constraint rather than a chore.

Backend first, frontend second

I built the backend as its own deployable service before touching the storefront UI. That meant I could test the actual logic — adding items, calculating totals, handling stock — with simple requests, long before there was anything to look at. By the time I started the frontend, I was building against an API that already worked, instead of guessing at one.

Deployment is part of the design

RedCart's frontend is deployed on Vercel; the backend is set up to run on Netlify functions or Render, depending on what's available. That flexibility wasn't an accident — early on I separated configuration from code so the same backend could run in more than one place without rewriting anything. It's a small habit, but it's the difference between a project that only runs on your laptop and one that's actually a product.

What I'd do differently

  • Write integration tests for the cart and checkout logic earlier — I caught two ordering bugs late that a test suite would have caught on day one.
  • Decide on error-handling conventions (what the API returns on failure) before building more than two routes, not after.
  • Document the Prisma schema relationships as I went, instead of relying on memory once the schema grew.

RedCart is still growing — next up is adding proper payment handling and order history. If you want to see it in action, the live demo is on the projects page.
